# Arcane EDR v0.8 GUI And MSI Research Baseline

Status: implemented release baseline for `v0.8.0`

Last researched: 2026-06-11

This note is the working baseline for the `v0.8.0` direction: Arcane is now a
coupled Windows service plus Windows GUI application, shipped with a WiX MSI
installer path. It is not a generic desktop-app essay. It is the design and
packaging posture Arcane should continue using as the GUI/MSI track matures.

## Executive Decision

Use a native Windows GUI, not a web shell, as the primary operator surface.
The preferred `v0.8.0` direction is:

- Windows service remains the always-on runtime authority.
- Windows GUI is the normal operator surface for configuration, validation,
  health, alerts, reports, maintenance, support bundles, and response rollback.
- Shared core code owns config parsing, policy parsing, validation, status
  models, report records, and redaction rules.
- Arcane keeps JSONL as its durable local evidence and state format. Do not add
  SQLite or another database for `v0.8.0`.
- Arcane uses one unified policy/config surface for allowlists, blocklists,
  remote endpoint trust, response gates, and alert tuning, backed by one scoped
  rule-engine pattern.
- MSI owns per-machine install, upgrade, repair, uninstall, service
  registration, shortcuts, install metadata, and default directories.
- GUI and CLI must not parse each other's human text. Any command used by the
  GUI should expose structured output.
- `ResponseMode=AlertOnly` remains the default and the GUI must make active
  response an explicit, evidence-backed operation.

Implemented in the first `v0.8.0` release candidate:

- WinUI 3 operator console under `src\ArcaneEDR.Gui`.
- MSI-oriented GUI publish script: `scripts\build-gui.ps1`.
- WiX v7 MSI authoring under `installer\` and `scripts\build-msi.ps1`.
- Service installation through MSI `ServiceInstall`/`ServiceControl`.
- Start menu shortcut to the GUI.
- License/warning UI that states local config/evidence are preserved by
  default.
- GUI Configuration reset-to-defaults flow with explicit warning checkbox and
  backup-before-reset behavior.

The strongest current framework choice for a new Windows-only GUI is WinUI 3
with the Windows App SDK. Microsoft describes WinUI as the recommended native
UI framework for new Windows apps, built on Windows App SDK and aligned with
Fluent Design. WPF remains a credible fallback if implementation risk needs to
be lower, but choosing WPF should be an explicit schedule decision, not the
default product direction.

## Repository Implications

The current project is compiled directly with .NET Framework `csc.exe` from
`scripts/build.ps1` and has no SDK-style solution. Do not big-bang this
repository into a new solution before the shared models and structured outputs
are stable. The staged direction is:

1. Keep the service build working.
2. Extract reusable config, policy, presentation, validation, and structured
   output models inside the current tree.
3. Add fixture tests around the CLI seams the GUI/MSI will depend on.
4. Introduce SDK-style projects once the boundaries are clear.

The eventual product solution should avoid growing the current flat source
layout indefinitely:

```text
src/
  ArcaneEDR.Core/
  ArcaneEDR.Service/
  ArcaneEDR.Cli/
  ArcaneEDR.Gui/
installer/
  ArcaneEDR.Installer/
  ArcaneEDR.Bundle/          optional, only if a bootstrapper is needed
```

`ArcaneEDR.Core` should contain the canonical config, policy, validation,
status, alert-presentation, report-summary, rule-engine, enrichment-provider,
and redaction models. The service, CLI, GUI, and installer custom checks should
all consume that shared model. Do not create a second GUI-only settings model.

Use a current LTS .NET target for new SDK-style projects. As of this research,
.NET 10 is LTS and supported until November 2028. A `v0.8.0` implementation can
still keep the service on .NET Framework temporarily if that reduces risk, but
the GUI should be built with a modern SDK-style project and the migration plan
should avoid permanent split-brain validation logic.

## GUI Framework Direction

### Recommended Stack

Use:

- WinUI 3.
- Windows App SDK stable channel only.
- C#/.NET SDK-style project.
- Fluent 2 design principles and WinUI controls.
- Self-contained GUI deployment for the first MSI milestone unless a
  bootstrapper is added.

Why:

- Microsoft positions WinUI as the recommended native UI framework for new
  Windows apps.
- Windows App SDK supplies modern WinUI, windowing, notifications, lifecycle,
  and deployment support across supported Windows versions.
- The Windows App SDK can also be used from WPF or Win32, which gives an escape
  hatch if part of the product must be staged.
- A self-contained GUI avoids requiring the operator to install a global
  Windows App SDK or .NET Desktop Runtime before Arcane can open. The tradeoff
  is larger install size and the need to ship updates when runtime dependencies
  need servicing.

Use WPF only if one of these becomes true:

- WinUI 3 deployment or build complexity blocks the MSI milestone.
- The GUI needs mature designer support more than it needs current Fluent
  fidelity.
- The project chooses a smaller, more utilitarian v0.8 GUI and defers visual
  polish.

Do not use Electron or a browser shell for the core v0.8 GUI. Arcane is a local
Windows safety product that needs native service status, UAC-aware workflows,
protected paths, file ACL clarity, installer integration, keyboard reliability,
and strong operator trust. A web shell would add footprint and update surface
without giving Arcane much benefit.

## Product Surface

The GUI should be an operations console, not a marketing page and not a
developer-only config editor.

Primary navigation:

- Overview: service state, protection posture, last poll, current mode,
  validation state, last report, and current alert volume.
- Alerts: recent alerts, grouped incidents, filters, local-only versus external
  status, and clear stale/current labeling.
- Policy: structured policy entries, preview results, disabled entries,
  suppress-external decisions, and sample preview tooling.
- Configuration: supported config sections, validation, secrets-by-environment
  guidance, and privacy controls.
- Maintenance: maintenance windows, baseline learning status, agent activity
  ledger, and expected operator activity markers.
- Reports: daily reports, AI compact-analysis status, payload preview, local
  archive, and destination health.
- Response: response mode, dry-run ledger, active response gating, rollback
  records, and manual rollback launch points.
- Support: support-bundle generation, redaction summary, version, docs links,
  install paths, and service diagnostics.
- Settings: app theme, update channel if supported later, telemetry/privacy
  preferences, and about/legal details.

The GUI should open to a useful overview, not a first-run pitch. On a fresh
install, it should show setup readiness with clear required steps:

- Configure local files or import existing config.
- Validate config.
- Install or confirm Sysmon if used.
- Install or confirm service.
- Start baseline learning.
- Keep external alerting and AI disabled until validation succeeds unless the
  operator intentionally enables them.

## UX And Aesthetic Standards

Arcane's UI should feel calm, native, and operational. It should not look like a
consumer antivirus scare screen, a SIEM dashboard, or a shiny SaaS landing page.

Principles:

- Keep the main frame quiet: navigation on the left, work surface on the right,
  compact status area at the top.
- Use status language from the existing confidence ladder: no confirmed
  compromise, needs review, watch or baseline, expected local context.
- Treat red as reserved for confirmed high-severity or action-required states.
  Use neutral, amber, blue, and green states for routine health and review.
- Prefer dense but readable layouts. Arcane users will scan repeated operational
  data; avoid oversized cards and decorative hero treatments.
- Use Fluent spacing and hierarchy. Related controls should be visually grouped
  through consistent spacing before adding borders or extra panels.
- Use Segoe UI Variable/WinUI type ramp defaults. Use large type only for the
  top-level overview title or major state, not every panel heading.
- Use Mica or solid surfaces sparingly. Mica is appropriate for long-lived app
  chrome; Acrylic belongs on transient flyouts or menus.
- Use icons for repeated navigation and commands where the metaphor is familiar,
  with labels or tooltips for security-specific actions.
- Make copy plain and bounded. The UI should say what Arcane saw, why it
  matters, and what the operator can do next.

Settings and controls:

- Binary settings should be toggles.
- Small mutually exclusive choices should be radio buttons or segmented
  controls.
- File/path settings should use text input plus browse.
- Thresholds should use numeric inputs, not vague sliders, unless the numeric
  value remains visible.
- Risky commands should use command buttons with an evidence summary, expected
  effect, and rollback path, not a generic "Are you sure?" dialog.
- Settings should validate immediately and save through the same config model
  used by `--validate-config`.

## Accessibility Requirements

Accessibility is a release gate, not polish.

Minimum requirements:

- Every interactive control has an accessible name.
- Form inputs use visible labels that map to UI Automation labels.
- Keyboard navigation reaches every command and preserves logical tab order.
- Composite controls support arrow-key navigation where expected.
- Alerts, validation errors, and status changes are exposed to assistive
  technology.
- Text remains readable under Windows display scaling and text scaling.
- High contrast themes are tested.
- Normal text meets contrast requirements and color is never the only signal.
- Tables and grids support keyboard selection, copyable cell content, and
  meaningful column names.
- No workflow requires pointer-only gestures.

Arcane-specific accessibility details:

- Alert rows should expose rule ID, severity, time, process, and current
  assessment in screen-reader-friendly order.
- Dangerous response commands must announce the target and response mode before
  activation.
- The app should provide copyable redacted summaries so the operator can move
  evidence into notes without exposing raw private paths or command lines.

## Security And Elevation UX

The GUI should run as standard user by default. Read-only status should remain
available without elevation wherever possible. Administrative work should be
per-task and explicit.

The Windows service must remain non-interactive. Microsoft service guidance
warns that services run in Session 0 and should not present UI to the signed-in
user. Arcane's service should collect, score, enrich, respond according to
policy, and write durable local evidence; the GUI should be a separate client
over structured files, structured CLI commands, or a tightly scoped local IPC
surface.

Recommended elevation model:

- GUI process manifest: `asInvoker`.
- Installer: per-machine MSI requiring elevation.
- Service install, service restart, Sysmon install, publish/restart, and
  response rollback: invoke constrained elevated operations.
- Prefer structured read-only CLI/JSON outputs for the first GUI milestone.
- Keep the existing scheduled-task admin bridge model for constrained elevated
  operations until a better local service operation API exists.
- If a service IPC API is added, use a local named pipe or equivalent with a
  tight ACL, documented access rights, structured commands, request IDs, audit
  logging, input validation, bounded payload sizes, and no generic shell
  execution.

Do not make the GUI permanently elevated. Microsoft UAC guidance favors running
apps with the least privilege needed and requiring elevation only when the task
needs it. Permanent elevation would also make the GUI a more attractive target.

Risky action UX:

- Show the triggering evidence.
- Show the exact target.
- Show expected effect.
- Show rollback path.
- Write a local audit record.
- Keep active response off unless the operator explicitly changes response
  mode.

Avoid generic confirmation patterns for dangerous work. They train clicks and
do not help the operator reason about the action. Use a short review panel that
requires the operator to understand the target and rollback.

## Configuration And Data Model

The GUI should edit the same local config files Arcane already treats as
operator-owned state, but it should do so through structured APIs:

- `config\ArcaneEDR.config`
- `config\Deployment.config`
- `config\arcane-policy.json`
- future GUI metadata under a documented local JSON or JSONL state path

Policy and local state rules:

- `arcane-policy.json` is the unified allowlist, blocklist, response gate,
  remote endpoint, and alert tuning file.
- Rule evaluation uses explicit scopes such as `alert`, `remote_endpoint`,
  `response`, and future `report` so one engine pattern can expand without
  mixing unrelated match fields.
- JSONL remains the append-only evidence format for alerts, incidents, response
  ledgers, activity ledgers, external-alert sinks, reports, and support context.
- Do not introduce SQLite or another embedded database for `v0.8.0`.
- Any future indexing/cache layer must be rebuildable from JSONL evidence and
  must not become the canonical record.

Needed implementation work before a full GUI:

- A real parser/writer that preserves comments where practical or emits a
  clean canonical file.
- Structured validation result model with severity, key, source file, line
  number when available, remediation, and whether restart is required.
- Structured policy inspect and policy preview output.
- Structured alert-volume and incident-summary output.
- Structured service health model.
- Shared alert presentation model for title, source, remote endpoint, country,
  company, ASN, domain, and enrichment source.
- Explicit config ownership boundaries: install examples are immutable,
  operator config is mutable, logs/evidence are preserved.

The GUI should never store provider secrets in tracked files or installer
properties. It may store environment variable names, and it may help the
operator open Windows environment-variable UI or run documented commands, but
secret values remain in the appropriate Windows environment scope or another
explicit secret store.

## Install And Runtime Paths

`v0.8.0` should choose canonical product paths before authoring the MSI.

Preferred Windows packaging shape:

- Binaries and immutable docs/examples: `%ProgramFiles%\Arcane EDR\`
- Machine config, state, logs, reports, bundles: `%ProgramData%\Arcane EDR\`
- Optional high-signal evidence export: configurable, with current `C:\Security`
  deployments handled by explicit migration notes.

Resolved product direction: MSI-owned Arcane installs product files under
`%ProgramFiles%\Arcane EDR\`. Existing `C:\Applications\ArcaneEDR` deployments
are old source/script install state, not a supported runtime location after MSI
cutover verification passes.

Default uninstall behavior:

- Remove installed binaries, service entries, Start menu shortcuts, installer
  metadata, and Arcane-owned scheduled tasks.
- Preserve config, logs, reports, incident history, response ledger, and
  support-bundle history by default.
- Offer a clearly labeled purge path only when the operator explicitly chooses
  to remove local evidence and config.

## MSI Packaging Direction

Use WiX Toolset for MSI authoring. As of this research, WiX v7 is current in
the FireGiant/WiX documentation and supports SDK-style MSBuild projects and
the `wix.exe` .NET tool. Pin the WiX major/minor in repo tooling and re-check
license/EULA and current release status before shipping `v0.8.0`.

Use MSI as the core installer because Arcane needs per-machine service
installation, protected install locations, repair/uninstall semantics, upgrade
identity, and admin-owned machine state. A Burn bundle can be added later if
Arcane needs to chain prerequisites such as a .NET runtime or Windows App SDK
runtime installer. Do not require a bundle if the MSI can be self-contained.

MSI authoring rules:

- Per-machine install only for `v0.8.0`.
- Single product identity with stable upgrade identity.
- Major upgrade support from the first MSI release.
- Do not add backward-compatibility installer aliases unless the operator asks.
- Put service executable in a manually authored component.
- Use `ServiceInstall` and `ServiceControl` for the Arcane service instead of
  installing the service through ad hoc shell commands.
- Use service configuration authoring for delayed auto start, failure actions,
  service SID, and privileges where supported.
- Use standard MSI tables/actions for files, shortcuts, registry, services, and
  environment changes wherever possible.
- Avoid custom actions. If custom actions are unavoidable, make them deferred,
  no-impersonate only when system changes require it, paired with rollback, and
  covered by install/uninstall/repair/upgrade tests.
- Never use nested/concurrent MSI installations.
- Do not write secrets to MSI properties, logs, registry, or config files.
- Provide silent install and silent uninstall command lines.
- Run MSI validation before first install attempt and in CI.
- Sign the MSI and every executable payload intended for release.

Upgrade rules:

- Increment one of the first three MSI version fields for every upgrade that
  should replace an installed product.
- Keep upgrade identity stable across versions that should upgrade each other.
- Block downgrades with a clear message.
- Test `v0.8.0 -> v0.8.1` before shipping `v0.8.0`, even if `v0.8.1` is only a
  dummy build.
- Preserve mutable config and evidence unless the operator explicitly purges.
- Restart or stop the service in a predictable sequence and surface that
  behavior in installer UI and docs.

Repair rules:

- Repair restores missing binaries, shortcuts, service registration, and
  install-owned examples/docs.
- Repair does not overwrite operator config or logs.
- Repair re-runs validation if possible and leaves a local installer log path
  for troubleshooting.

Rollback rules:

- Treat rollback as a first-class test scenario.
- Failed install should not leave a broken service entry.
- Failed upgrade should either leave the old working product or fail loudly with
  clear recovery steps.
- Any custom system mutation outside standard MSI tables needs a rollback
  counterpart.

Signing:

- Release MSI, bundle if present, service EXE, GUI EXE, helper EXEs, DLLs, and
  uninstall helpers should be Authenticode signed.
- Use timestamping.
- Self-signed certificates are for local development only.
- Unsigned public MSI/EXE distribution should be treated as not ready.

## Installer UI

The MSI UI should be boring in the best way.

It should include only:

- Welcome.
- Install location if configurable.
- Existing config detected summary.
- Preserve local evidence/config notice.
- Explicit "Reset config to defaults" checkbox when existing config is found.
  It must be unchecked by default and paired with warning copy that local
  recipients, provider settings, policy tuning, baseline choices, and
  machine-specific paths may be replaced.
- Optional launch GUI after install.
- Progress.
- Completion with "open Arcane EDR" and "view install log" options.

Do not ask for Brevo, SMTP, webhook, AI provider, API key, or recipient secrets
in the MSI. The GUI can guide that after install, with environment-variable
names and validation.

MSI behavior should hand-hold but preserve operator control:

- Clean install lays down baseline defaults and examples.
- Upgrade and repair preserve local config and JSONL evidence by default.
- Reinstall with existing config warns clearly and defaults to preserving it.
- Reset-to-defaults writes backups or makes the replacement path explicit.
- Purging config/evidence is a separate, explicit uninstall or maintenance
  choice, not normal upgrade behavior.

## Service And GUI Alignment

Every service feature intended for normal operators should have a GUI surface:

- Configuration key.
- Validation rule.
- Help copy.
- Restart requirement.
- Privacy note if it can leave the machine.
- Installer/update consideration if it affects files, services, tasks, event
  logs, permissions, or prerequisites.
- Test coverage for config -> validate -> service behavior -> GUI display.

Features without GUI coverage are allowed only when they are internal, debug
only, or explicitly scoped out. The roadmap should mark those exceptions.

## Test Matrix For v0.8.0

Minimum GUI tests:

- Launches as standard user.
- Launches without service installed and explains setup state.
- Shows service running/stopped/unknown state.
- Shows validation success and validation errors.
- Edits a harmless config setting and validates it.
- Refuses invalid config with precise location and remediation.
- Shows recent alerts without exposing raw private fields by default.
- Preserves keyboard navigation across all main views.
- Works in high contrast mode.
- Works at common Windows scaling values.

Minimum installer tests:

- Clean install on a fresh Windows VM.
- Install over existing source-ZIP deployment if migration is supported.
- Upgrade `v0.8.0` to a newer build.
- Repair after deleting a binary.
- Uninstall preserves config/logs by default.
- Explicit purge removes only documented Arcane-owned mutable state.
- Failed install rollback.
- Failed upgrade rollback.
- Silent install and uninstall.
- Non-admin install attempt gives a clear elevation path.
- MSI validation has no unresolved ICE errors.
- Installed service starts, stops, restarts, and recovers as documented.
- GUI starts after install and validates the installed service.

## Implementation Sequence

1. Extract shared config, policy, rule-engine, enrichment-provider,
   presentation, validation, and redaction models inside the current service
   tree.
2. Add fixture tests for the CLI seams the GUI/MSI will depend on: build,
   version, validation, policy inspect, policy preview, and report preview.
3. Add structured JSON outputs for validation, health, alert volume, incidents,
   reports, policy preview, agent activity, and response ledger.
4. Build a read-only WinUI GUI shell against structured outputs.
5. Add config editing for low-risk settings and validation.
6. Add policy editing and policy preview.
7. Add service maintenance, support bundle, and report surfaces.
8. Add response rollback surface while keeping active response gated.
9. Introduce WiX MSI for clean install/uninstall.
10. Add upgrade, repair, rollback, signing, and silent install support.
11. Replace ZIP-first install docs with MSI-first install docs for v0.8 and keep
    source/console flows as developer/operator alternatives.

## Open Questions Before Implementation

- Should the service migrate to modern .NET with the GUI, or should v0.8 keep
  service runtime risk low and migrate only the GUI/build system?
- Should the first MSI be MSI-only and self-contained, or should it ship with a
  Burn bootstrapper for prerequisites?
- What signing path will the project use for public release artifacts?

Resolved direction:

- The first GUI milestone talks to Arcane through structured CLI/JSON and local
  JSONL/state files. A locked-down named-pipe service API can be added later for
  operations that need live service mediation.
- JSONL remains the product storage model for `v0.8.0`; no SQLite/database.
- MSI-owned product files live under `%ProgramFiles%\Arcane EDR\`.
  `C:\Applications\ArcaneEDR` is not part of the supported MSI runtime path.
- Existing local config is preserved by default on upgrade, repair, reinstall,
  and uninstall. Reset or purge requires an explicit warned operator choice.
